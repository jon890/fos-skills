# Step 2 — 기술 스택 검증

## 목적

기존 스택으로 충분한지 확정하고, 새 의존·복잡도 도입은 정당화를 거친다.

## 필수 산출물

- 기존 스택으로 되는가 판단.
- **신규 외부 의존이 필요하면** — 대안 2~3개 비교(트레이드오프) + 사용자 논의. 정당화 없이 도입 금지.
- 복잡도(별도 서비스·queue·DB 등) 도입 시 근거.

## 누락 체크리스트

- "편하니까" 로 새 라이브러리를 슬쩍 넣지 않았나 — 비용(유지·빌드·학습) 짚었나.
- 기존에 같은 일을 하는 도구가 이미 있지 않나(중복 도입).
- 도입한 복잡도가 이 기능 규모에 맞나(과설계).

## 캐낼 모호점 (능동 발굴 → 사용자 확정)

- 새 의존 도입 여부가 애매한 곳 — 기존으로 되는데 새로 넣으려는가.
- 저장·비동기·캐시 같은 복잡도를 넣을지 말지 갈리는 지점.
- 버전·호환 제약이 불명확한가.

## 게이트 통과

위 모호점 **0** + 스택 확정(기존으로 충분 / 새 의존 X를 Y 근거로 도입) 사용자 동의 → 3단계.
