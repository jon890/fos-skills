# Step 1 — 구현 가능성 검증

**역할**: CTO. "이거 정말 되나, 뭘 재사용하나, 뭐가 위험한가"를 냉정하게 본다.

## 목적

기술적으로 구현 가능한지, 기존 코드에서 뭘 재사용하는지, 리스크·제약이 뭔지 확정한다.

## 필수 산출물

- 구현 가능 판단(가능/조건부/불가) + 근거.
- **재사용 지점** — 기존 코드·모듈·스키마 중 그대로 쓸 것.
- **리스크·제약** — 외부 의존·성능·권한·데이터 등 걸림돌.
- 불확실하면 즉시 사용자와 논의(추측 금지).

## 누락 체크리스트

- "될 것 같다"로 넘기지 않았나 — 재사용·리스크를 **구체적으로** 짚었나.
- 이 기능이 기존 결정(ADR)과 충돌하지 않나.
- 다른 세션·plan 이 같은 영역을 건드리고 있지 않나(동시성).

## 캐낼 모호점 (능동 발굴 → 사용자 확정)

명시 안 됐거나 해석이 갈리면 추측 말고 옵션+트레이드오프로 묻는다:

- 요구의 해석이 2개 이상인가 — 무엇을 만드는 건지 경계가 애매한가.
- 성공 기준이 불명확한가 — "됐다"의 정의는.
- 재사용이냐 신규 작성이냐 갈리는 지점.
- 이 변경의 범위(어디까지 손대나)가 열려 있나.

## 게이트 통과

위 모호점이 **0** 이고, 사용자가 "구현 가능·이 재사용·이 리스크"에 동의 → 2단계.
