## A. 부패 (Decay)

코드가 docs 와 맞는가. 제거된 엔티티가 docs 에 남아있지 않은가.

- 제거된 함수·컴포넌트·엔드포인트명이 docs 에 남아있다.
- DB 스키마와 스키마 문서의 테이블·필드가 불일치한다.
- 디렉터리·파일 경로가 실제와 다르다.
- "제거 완료" ADR 의 제거 대상이 여전히 코드에 존재한다.
- 미갱신 환경변수 표, 실재하지 않는 경로.

> 레포 특화 부패 검사(스키마 파일 경로·라우트·config 키 대조)는 오버레이의 grep 명령을 따른다.

## B. 과대화 (Bloat)

ADR·docs 가 구현 세부로 비대해졌는가.

**ADR 은 "기술적 의사결정" 만 담는다.** 구현 결과(컴포넌트 트리·파일 배치·함수 호출)는 코드에 있다.
ADR 이 답해야 할 질문은 *"무엇을 / 왜 / 다른 선택지는 왜 아닌가"* 뿐이다.

**최종 상태만 기록하고 중간 단계는 제거한다.**
ADR 은 "현재 이렇게 되어 있다" 와 "왜 이 결론에 도달했는가" 만 남긴다.
여러 번 이터레이션한 중간 작업·변경 항목 목록은 삭제한다. 진화 과정이 의미 있으면 한 문장으로 응축한다.

검출 대상:

- ADR 본문 내 긴 코드 블록 (구현 예시)
- 파일 경로 3개 이상 나열
- 변수·함수·프롭 이름을 줄 단위로 나열하는 표
- "사용 예시" 섹션에 호출 코드 복붙
- CLAUDE.md 스택 규칙을 반복하는 문장
- "새 디렉터리 구조", "변경 항목 목록", "레거시 삭제 목록" 같은 작업 내역

> 정당한 예외: **단일 결정 표현에 필요한 최소한의 식별자**(attribute 이름 1~2회, 타입명 1회 등)는 허용한다.

## C. 의사결정 추론성 (Clarity)

ADR 이 "왜" 를 담고 있는가. "결정 / 맥락 / 대안 기각" 구조가 있는가.

검출 대상:

- "결정" 만 있고 "맥락·이유·대안" 이 없는 ADR
- 맥락이 "요구사항 추가" 같은 순환 설명
- 대안 기각이 없거나 "기각" 만 있고 근거가 없음

## D. 중복 (Duplication)

같은 내용이 여러 docs 에 본문으로 반복되는가.

검출 대상:

- 같은 필드 목록이 ADR 과 스키마 문서 양쪽에 있음
- 흐름 요약이 PRD 와 flow 문서 양쪽에 있음
- CLAUDE.md 스택 규칙이 아키텍처 문서에 복사됨

원칙: **정의는 1곳, 다른 문서는 링크 또는 짧은 참조.**

## E. 자명성 (Self-evidence) — ADR 전용

**코드·설정 파일만 봐도 아는 결정은 ADR 로 남기지 않는다.** ADR 의 가치는 "코드를 봐도 알 수 없는 맥락" 에 있다.

폐기 후보 유형:

| 유형 | 예시 | 근거 |
|---|---|---|
| 라이브러리·패키지 선택 | "X 사용", "Y 사용" | 의존성 파일로 자명 |
| 폴더·디렉터리 구조 결정 | "X 폴더 분리" | 디렉터리 트리로 자명 |
| 단순 마이그레이션 기록 | "X 컬럼 제거", "Y 를 Z 로 rename" | 스키마와 git log 로 자명 |
| 일반 프로그래밍 원칙 | "중복 코드는 공통 함수로 추출" | 상식 |
| 환경 설정 | "개발 포트 분리", "Compose 구성" | 설정 파일로 자명 |

유지 기준 (아래 중 최소 하나에 해당해야 보존):

1. **라이브러리 고유 함정** — 문서에 없거나 직관에 반하는 API 특성
2. **실험 결과** — A/B 비교로 도출된 근거 (수치 포함)
3. **대안 기각 근거** — "왜 이 선택이 아닌가" 가 미래 논의에서 재발할 여지가 있을 때
4. **정책·규칙 결정** — 팀 합의로 정해지고 준수해야 할 규율
5. **비용·성능 트레이드오프 근거** — 수치가 있는 결정

자명성 판정 질문 (세 개에 모두 YES 면 폐기 대상):

- 이 ADR 이 없어도 코드·설정·git log 로 같은 정보를 얻을 수 있는가?
- "왜" 가 1~2문장으로 정리되지 않고 그냥 "그렇게 되어 있다" 수준인가?
- 다른 프로젝트에서도 일반적으로 하는 선택인가?

## F. 가독성 (Readability) — 모든 docs

마크다운 작성 규칙 위반을 점검한다.
정책 본문은 전역 `~/.claude/rules/markdown-readability.md` 와 `korean-style.md` 에 단일 소스가 있다 — 본 축은 검출 휴리스틱만 둔다.
레포 CLAUDE.md 에 자체 가독성 규칙이 있으면 그쪽을 우선한다.

대상: 모든 `*.md` 문서. 코드 블록·표·디렉터리 트리는 미적용.

검출 휴리스틱:

- **semantic line break 위반**: 한 줄에 `. ` / `? ` / `! ` 가 2회 이상 (한 문장 = 한 줄 원칙 위반)
- **인라인 나열**: `①②③` 같은 원문자, 또는 한 줄에 `/` 3개 이상의 병렬 나열, 또는 `A + B + C` 인라인 연결
- **괄호 중첩**: `\([^)]*\([^)]*\)` — 2겹 이상
- **동치·인과 압축**: 한 단락에 `=` 또는 `→` 가 2회 이상
- **긴 줄**: 한 줄 200자 초과 (화면 폭 기준, 한글은 문자 수로 판단)
- **명사형 종결**: paragraph 평문 문장이 "필요.", "확보.", "미확정." 같은 명사형으로 끝남
- **`~` 취소선 함정**: 한 단락·한 줄에 `~` 가 짝수개 (범위 표기 `60~100` 에서 자주 발생, `<del>` 로 렌더됨)
- **`§` 사용**: 섹션 기호는 직관이 어려움
- **Bold + 괄호**: `**텍스트(영문)**` 형태는 일부 파서에서 bold 가 안 됨 (`**텍스트**(영문)` 로)

## 스킬 SKILL.md 의 6축 변형

스킬은 AI 에이전트가 호출 시점에 컨텍스트로 로드한다 — 장황하면 토큰을 낭비하고 핵심 지시가 묻힌다.
ADR 과 다른 자명성 기준을 적용한다.

| 축 | 스킬 적용 |
|---|---|
| **B 과대화** | 사고 사례 길게 서술(왜·언제·증상), 자명한 부연, 다른 스킬과 중복되는 일반 원칙은 모두 bloat. 핵심 지시 + 검증 명령 + 1줄 사유로 응축 |
| **C 추론성** | 지시만 있고 "왜 이 가드가 필요한지" 단서 1줄이 없으면 미래 에이전트가 가드를 우회 |
| **D 중복** | 같은 가드·원칙이 여러 스킬에 반복되면 공용 `_shared/` 로 추출 |
| **자명성**(E 변형) | 에이전트의 일반 행동 디폴트로 추론 가능한 지시(예: "tool 결과 확인 후 진행")는 제거 |

검출 신호:

- 한 섹션에 사고 사례 2개 이상 나열되면 하나로 응축한다. 진화가 의미 있으면 "X 였다가 Y 를 거쳐 Z 로 수렴" 한 문장으로.
- 같은 규칙을 "필수" / "강제" / "반드시" 세 번 반복한다.
- 한 항목에 코드 블록·인용·표가 모두 등장한다 (보통 1~2개로 충분).
- 다른 스킬에 동일한 cwd 격리·scope 가드가 다른 표현으로 중복된다.
