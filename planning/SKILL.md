# planning

새 기능이나 변경사항을 구현하기 전, 모호함을 모두 해소하고 문서를 정비한 뒤 실행(build)으로 넘기는 8단계 설계 워크플로.

이 스킬은 **여러 레포가 공유하는 단일 코어**다 (`~/personal/fos-skills/planning`, 글로벌 `~/.claude/skills/planning` 로 symlink). 워크플로 개선은 여기 한 곳만 고치면 전 레포에 반영된다. 레포마다 다른 부분(도메인 단계·docs 컨벤션·검증 경로·실행 명령)은 아래 오버레이로 주입한다.

## 레포 오버레이 로딩 (필수 첫 단계)

작업 시작 시, 현재 레포에 오버레이 파일이 있으면 **먼저 읽고** 그 지시를 코어보다 우선한다:

- 경로: `<repo-root>/.claude/planning-overlay.md`
- 오버레이가 정의하는 것: 도메인 유형(UI/CLI/backend), 단계 3~6의 도메인 변형, docs 컨벤션(문서 목록·ADR 위치), 검증(common-pitfalls 경로·추가 패턴), plan 네이밍·branch 규칙, 실행 핸드오프 명령.
- 오버레이가 **없으면** 코어 기본값으로 동작한다 (도메인 중립).

오버레이는 코어를 *덮어쓰는* 게 아니라 *채운다* — 코어가 뼈대, 오버레이가 레포 살점.

## 핵심 원칙

- **모호함 제로**: 각 단계에서 조금이라도 모호하면 반드시 사용자와 논의. 넘어가지 않는다.
- **AI 에이전트 관점**: 최종 문서는 AI 에이전트가 읽고 phase 로 구현할 수 있을 정도로 명확해야 한다.
- **간결한 문서**: 컨텍스트 낭비 금지. 의사결정 의도는 보존하되 코드로 확인 가능한 구현 상세는 뺀다.
- **Critic 반복 지적 사전 해소**: 오버레이가 common-pitfalls 파일을 지정하면, task 작성 시 그 패턴을 self-check 해 critic 이 같은 지적을 반복하지 않게 한다.
- **선택지 제시는 질문 도구로**: 옵션 중 하나를 고르게 할 때는 현재 에이전트의 구조화 질문 도구(Claude Code 는 `AskUserQuestion`)를 쓴다. 추천안은 첫 번째 + label 끝 `(추천)`. 글로 늘어놓는 long-form 옵션 비교 금지.
- **질문 규율**: 위 도구 사용법에 더해 세 가지를 지킨다.
  - **깊은 모호함은 하나씩**: 상호 의존적이거나 깊이 파야 하는 결정은 한 번에 하나씩 묻는다. 여러 개를 동시에 던지면 답이 얕아진다. 서로 독립적인 결정은 질문 도구로 묶어 물어도 된다.
  - **선택지엔 트레이드오프 + 추천**: 옵션을 낼 때는 트레이드오프를 붙이고 추천안을 먼저 둔다. 열린 질문 하나만 덜렁 던지지 않는다.
  - **질문은 결정에만, 사실은 직접 조회**: 환경(파일·git·도구)에서 확인 가능한 *사실*은 사용자에게 묻지 말고 직접 조회한다. 사용자에게는 사람만 답할 수 있는 *결정*만 묻는다.

## 실행 절차

사용자가 `/planning <기능 설명>` 을 호출하면 아래 단계를 **순차적으로** 진행. 각 단계는 사용자의 확인을 거친 후 다음 단계로. 규모가 작은 기능은 여러 단계를 합치거나 생략할 수 있다 (아래 "단계 건너뛰기").

### 0단계: 보정 (선택)

본격 질문 전, 질문의 깊이와 강도를 사용자에 맞춘다. 매번 하지 않고 사용자의 의도가 불명확할 때만 수행한다.

- **지식 수준** (New / Working / Expert): Expert 에게 기초를 설명하느라 낭비하지 않고, 학습자에게 과도한 압박을 주지 않도록 설명 깊이를 맞춘다.
- **압박 강도** (Light / Standard / Hard):
    - Light — 모호함만 해소한다.
    - Standard — 가정에 도전한다.
    - Hard — 실패 모드와 롤백 가능성까지 파고든다.

작은 기능이거나 사용자가 이미 방향을 명확히 줬으면 이 단계를 생략한다.

### 1단계: 구현 가능성 검증

**역할**: CTO

- 기술적으로 구현 가능한지 검증.
- 기존 코드베이스에서 재사용 가능한 부분 식별.
- 리스크·제약사항 도출.
- 모호한 부분이 있으면 즉시 사용자와 논의.

### 2단계: 기술 스택 검증

- 기존 스택으로 충분한지 확인.
- 새 라이브러리 도입이 필요하면 대안 비교 + 사용자와 논의. **신규 외부 의존성은 비용이 크다** — 정당화 없이 도입 금지.
- 복잡도(별도 서비스·queue·DB 등)를 도입한다면 정당한 근거가 있는지 검증한다 — 필요하면 도입하되, 근거 없이 늘리지 않는다.

### 3단계: 호출/사용자 흐름 검증

**역할**: 시니어 워크플로/UX 디자이너

- 새 기능이 어떻게 호출·사용되는지 명확화. 정상 흐름 + 에러 흐름 + 빈 상태 + 권한·잠금 충돌 같은 엣지 케이스 점검.
- **도메인 변형은 오버레이 참조** — UI 레포는 화면 전환·사용자 액션, CLI 레포는 명령 인자/플래그 조합, backend 는 호출 시퀀스로 구체화.

### 4단계: 인터페이스 설계

- 사용자·호출자가 마주하는 표면을 설계.
- **도메인 변형은 오버레이 참조** — UI 는 화면별 정보·컴포넌트 구조, CLI 는 명령 시그니처·출력 포맷, backend 는 엔드포인트별 요청/응답 스키마. (backend 는 4·5단계가 겹치면 병합.)

### 5단계: API/함수 설계

- 필요한 엔드포인트·함수 목록과 요청/응답 스키마 초안.
- 기존 엔드포인트·함수 재사용 가능 여부.
- **도메인 변형은 오버레이 참조.**

### 6단계: 데이터/코드 구조 설계

**역할**: 데이터 모델러 + CTO

- 새 config·스키마·runtime 상태가 필요한가? 위치와 정확한 스키마.
- 코드 아키텍처(디렉터리·레이어 분리), 기존 패턴과의 일관성.
- 산출물(report·JSON 등)의 위치·명명 규칙.
- 모호한 설계 결정은 사용자와 논의.

### 7단계: docs 영향 종합 + 기술 결정 기록

- 이번 기능에서 내린 기술 결정을 목록화하고 각 결정의 "왜"를 기록.
- 어떤 문서를 갱신할지, ADR 을 어디에 어떻게 남길지는 **오버레이의 docs 컨벤션 참조**.
- **단일 소스 원칙**: 같은 정보를 두 문서에 적지 않는다. 참조가 필요하면 링크.
- 자명한 결정(자료를 보면 바로 아는 것)은 ADR 로 남기지 않는다.

### 8단계: task 파일 생성 + 커밋

`tasks/plan{N}-<kebab-slug>/` 아래 `index.json` + `phase-*.md` 작성. 상세 규칙은 [`task-create.md`](./task-create.md) 참조.
docs 반영과 task 생성을 마친 뒤 커밋한다 (커밋 분리·핸드오프 명령은 오버레이·아래 "완료 후" 참조).

## 의사결정 누적 → 논의 완료 시 docs 반영

논의 중에는 각 단계의 결정을 대화 컨텍스트에 진행 요약으로 누적한다. 최종 docs 에 즉시 쓰지 않는다 — 중간 결정은 뒤 단계에서 바뀔 수 있어, 즉시 쓰면 재작성 반복과 미확정 내용 혼입이 생긴다.
논의가 끝나면 누적된 최종 결정만 docs 에 한 번에 반영한다.

## self-review (제출 전 fresh eyes, 필수)

task 전체를 새로운 눈으로 한 번 훑는다. critic 이 아니라 본인이 도는 체크리스트다.

1. **placeholder 스캔**: `TBD` / `TODO` / '나중에' / 빈 섹션이 남아 있나. 있으면 실제 내용으로 채운다.
2. **모순 스캔**: phase 간·docs 간 상충하는 서술이 없나.
3. **식별자 일관성**: 앞 phase 에서 정한 함수·타입·필드명을 뒤 phase 가 똑같이 쓰나. phase-01 의 `clearLayers()` 를 phase-03 이 `clearFullLayers()` 로 쓰면 버그다.

발견 즉시 인라인 수정. 재검토는 불필요 — 고치고 넘어간다.

## task 검증 (생성 직후, 필수)

task 파일 작성 직후, 사용자 보고 + git commit 전에 검증한다. 상세 절차·패턴은 [`task-create.md`](./task-create.md) 의 "task 검증" 섹션 참조. 요약:

- **자동 검출 5 패턴** — `scripts/verify-task.sh plan{N}-{slug}` 실행. 출력 0 줄이면 통과.
- **위반 처리** — 수정 / skip / 면제 중 선택. AI 가 임의로 자동 수정하지 않고 질문 도구로 확인받는다.
- **사람 판단 4 패턴** — grep 불가. 직접 self-check.

## 완료 후 (필수 수행 절차)

단계가 끝나면 항상 아래를 수행 — 사용자의 별도 지시를 기다리지 않는다.

1. **docs 반영 완료 확인** — 오버레이가 지정한 문서에 이번 결정이 기록됐는지 점검.
2. **task 파일 생성 완료 확인** — `tasks/plan{N}-<slug>/` + index.json + phase 파일들.
3. **task 검증 실행** — `verify-task.sh` 5 패턴 0 건 + 사람 판단 4 패턴 self-check + self-review. 위반 시 질문 도구 흐름.
4. **branch 확인** — 기본은 `main`. 레포 branch 정책은 오버레이 참조.
5. **git commit** — docs 변경과 task 파일을 커밋. 커밋 분리 방식은 레포 규칙(오버레이) 우선.
6. **git push** (원격 정책이 허용하면).
7. **사용자 보고 + 실행 안내** — 오버레이가 지정한 실행 명령(예: `/build-with-teams`)을 안내.

**실제 phase 실행은 사용자가 명시적으로 실행 명령을 호출할 때 시작.** planning 은 task 생성 + push 까지만 책임진다.

### 예외

- **docs 변경 없음 + task 없음** (논의만): commit/push 생략, "task 생성할 규모 아니라 기록 없이 종료" 고지.
- **force push 필요**: 금지. 새 커밋 생성으로 대체.
- **원격 branch protection 으로 main 직접 push 차단**: PR 경로로 우회.

## 단계 건너뛰기

규모에 맞춰 단계를 합치거나 생략한다 — 작은 변경은 여러 단계를 하나로 묶고, 큰 변경은 전체를 밟는다. 어느 단계를 생략할지는 고정 규칙이 아니라 "모호함 제로" 가 유지되는 선에서 판단한다. 단 7단계(docs 영향 분석)만은 규모와 무관하게 항상 수행해 문서 drift 를 막는다.

## plan 네이밍 규칙

### 번호 충돌 방지 (필수)

plan/ADR 번호를 부여하기 전에 반드시 기존 번호를 확인한다. 다른 세션이나 열린 PR 이 번호를 점유했을 수 있다.

```bash
# cwd: <repo root>
ls tasks/ | grep "plan{후보번호}"
gh pr list --state open --json number,headRefName,title --jq '.[] | "\(.headRefName) \(.title)"'
```

다음 가용 번호를 사용. ADR 번호 확인 방식은 오버레이의 docs 컨벤션 참조.

### 서브넘버 규칙

비슷한 성격의 후속 작업은 같은 번호에 서브넘버를 붙여 묶는다 (예: `plan003` → `plan003-2`).
묶기 기준: 동일 기능 확장 / 동일 도메인 후속. 별도 번호: 서로 다른 도메인 / 독립 실행 가능.

## 파일

## 의도적으로 안 하는 것

- **이 세션에서 phase 실행**: planning 은 task 파일 생성까지만.
- **레포 특화를 코어에 하드코딩**: 도메인 단계·docs 컨벤션·검증 경로는 오버레이로만.
- **chat 에서 끝나는 결정**: 결정은 즉시 docs 에. chat 은 휘발.
