# docs-check

레포의 docs 를 **AI 에이전트가 최소 컨텍스트로 의사결정 의도를 재구성할 수 있는 상태**로 유지하기 위한 감사 스킬.

docs-check 는 **발견 전용**이다 — 감사해서 리포트를 만들고, 수정은 사용자 승인 후에만 한다.

## 레포 오버레이 로딩 (필수 첫 단계)

작업 시작 시, 현재 레포에 오버레이 파일이 있으면 **먼저 읽고** 그 지시를 코어보다 우선한다:

- 경로: `<repo-root>/.claude/docs-check-overlay.md`
- 오버레이가 정의하는 것:
  - docs 구조와 문서 목록 (어떤 docs 가 어디에 있는지)
  - 코드 ↔ docs 부패 검사용 grep 명령 (스키마 파일·라우트·config 경로는 레포마다 다르다)
  - 레포 전용 docs-verifier 에이전트 이름 (있을 때만)
  - common-pitfalls·docs 컨벤션 경로
  - 실행 주기·핸드오프 명령
- 오버레이가 **없으면** 레포 `CLAUDE.md` 의 docs 관련 규칙을 참조하고, 그래도 없으면 코어 기본값(도메인 중립)으로 동작한다.

오버레이는 코어를 *덮어쓰는* 게 아니라 *채운다*.

## 핵심 철학

**ADR 은 "현재 유효한 기술 의사결정" 의 참고자료이지 프로젝트 역사책이 아니다.**
AI 에이전트가 grep 해 능동 참조할 가치가 있으려면, 문서의 **대부분**이 "코드만 보고는 알 수 없는 WHY" 여야 한다.
자명한 결정·마이그레이션 기록이 쌓이면 시그널 대비 노이즈 비율이 떨어져 결국 아무도 안 읽는다.
AI 에이전트는 코드만큼 docs 를 신뢰하므로, 아래 6축(부패·비대·추론 불가·중복·자명·가독성) 중 하나라도 무너지면 잘못된 코드 생성으로 이어진다.

## 검증 6축

| 축 | 정의 |
|---|---|
| **A. 부패**(Decay) | 코드가 docs 와 맞는가. 제거된 엔티티가 docs 에 남아있지 않은가 |
| **B. 과대화**(Bloat) | ADR·docs 가 구현 세부(코드 블록·경로 나열·호출 방법)로 비대해졌는가 |
| **C. 추론성**(Clarity) | ADR 이 "결정 / 맥락 / 대안 기각" 구조로 왜를 담고 있는가 |
| **D. 중복**(Duplication) | 같은 내용이 여러 docs 에 본문으로 반복되는가 |
| **E. 자명성**(Self-evidence, ADR 전용) | 코드·설정 파일만 봐도 아는 결정을 ADR 로 남기고 있는가 |
| **F. 가독성**(Readability, 모든 docs) | 마크다운 작성 규칙(semantic line break·`~`·`§`·Bold+괄호 등) 위반이 있는가 |

각 축의 검출 대상·표·휴리스틱, 스킬 SKILL.md 전용 변형 기준은 [`references/six-axis.md`](references/six-axis.md) 참조.

## Hybrid 실행 모델

docs-check 는 두 종류의 검사를 서로 다른 방식으로 돌린다.

- **가벼운 정적 검사 — 메인이 직접**: grep·wc·diff 로 즉시 판정되는 것.
  ADR Index sync, 30줄 초과 bloat, 깨진 파일 참조, 금지 용어, `~`·`§`·Bold+괄호 같은 문체 정적 패턴.
  이들은 결정적이라 위임 오버헤드가 아깝다.
- **무거운 의미 검사 — read-only 에이전트에 위임**: 사람 판단이 필요한 것.
  orphan 문서, cross-link 후보, README ↔ 실제 파일 정합, 5축(A~E)의 의미 판정.
  여러 영역이면 병렬 위임한다.

**작성과 검증 분리 원칙**: 6축 의미 판정은 가능한 환경에서는 read-only 검증 에이전트에 위임한다.
작성한 주체가 같은 맥락에서 자기 검증하면 편향으로 결함을 놓친다 (하네스 원칙).

- 레포에 전용 docs-verifier 에이전트가 있으면(오버레이가 이름 지정) 그것을 우선 쓴다.
  그 에이전트 본문이 검증 항목·grep 명령·도메인 지식의 단일 소스다 — 메인이 grep 을 베끼면 정의가 두 곳으로 갈라진다.
- 전용 에이전트가 없으면 범용 read-only 리뷰 에이전트(예: `verifier`)에 위임하거나, 위임이 불가한 환경이면 메인이 직접 6축을 점검한다.
- 위임 시에도 수정 권한은 주지 않는다 — 검증 에이전트는 read-only. 메인이 회신을 받아 사용자 승인 후 수정한다.

## 실행 절차

### 1. 대상 파일 수집

오버레이·CLAUDE.md 가 지정한 docs 목록을 수집한다. 지정이 없으면 레포의 `docs/**/*.md`·`*.md`·`.claude/skills/*/SKILL.md` 를 훑는다.

### 2. 가벼운 정적 검사 (메인 직접, fail-fast)

의미 검사 전에 정적 검사를 먼저 돌려 결정적 위반을 잡는다. 아래는 도메인 중립 검사다 — 레포 특화 grep(스키마·라우트 대조)은 오버레이 명령을 실행한다.

**ADR Index 동기화** — 본문 ADR 번호가 Index 에 모두 있는가:

```bash
# cwd: <repo root> — ADR 위치는 오버레이 참조 (예: docs/adr/)
BODY=$(grep -rhoE '^## ADR-[0-9]+' <ADR_DIR>/*.md 2>/dev/null | grep -oE 'ADR-[0-9]+' | sort -u)
INDEX=$(grep -oE 'ADR-[0-9]+' <ADR_DIR>/INDEX.md 2>/dev/null | sort -u)
diff <(echo "$BODY") <(echo "$INDEX") && echo "OK: ADR Index synced"
```

**ADR bloat** — 파일·항목이 30줄을 넘으면 기능 명세 변질 의심:

```bash
for f in <ADR_DIR>/*.md; do
  size=$(wc -l < "$f" | tr -d ' ')
  [ "$size" -gt 30 ] && echo "BLOAT: $f ($size lines > 30) — 슬림화 검토"
done
```

**문체 정적 패턴** — `~` 짝수개·`§`·Bold+괄호(F 가독성):

```bash
for f in $(git ls-files '*.md'); do
  awk -v F="$f" '{ n=gsub(/~/,"~"); if (n>0 && n%2==0) print F":"NR": ~ 짝수개("n") — 취소선 렌더 위험" }' "$f"
  grep -nP '\x{00A7}' "$f" && echo "  ↑ $f: § 섹션 기호"
  grep -nE '\*\*[^*]*\([^)]*\)\*\*' "$f" && echo "  ↑ $f: Bold+괄호 — **텍스트**(영문) 로"
done
```

정적 검사 결과는 리포트 Critical 섹션으로 통합한다.

### 3. 무거운 의미 검사 (위임)

위 "Hybrid 실행 모델" 에 따라 read-only 에이전트에 6축(A~E) 의미 판정과 orphan·cross-link·README 정합을 위임한다.
여러 영역이면 병렬로. 위임 불가 환경이면 메인이 직접 점검한다.

### 4. 리포트 작성

발견 항목 포맷:

```markdown
#### [축] <파일> ADR-XXX — {한 줄 요약}
- **위치**: line N~M
- **문제**: {예: 15줄 코드 블록 — bloat}
- **근거**: {정적 검사 결과 / CLAUDE.md 규칙 / 실측 / 중복 문서}
- **제안**: {예: "결정/맥락/대안 기각으로 압축, 스니펫 제거"}
- **대안**: {유지 근거가 있으면 명시}
```

요약:

```markdown
## docs-check 결과 — YYYY-MM-DD

### Summary
- 검사 파일: N개 / ADR: M개
- 정적 검사: Index sync X / Bloat Y / 문체 Z
- 의미 6축: 부패 A / 과대화 B / 추론성 C / 중복 D / 자명성 E / 가독성 F

### Critical (즉시 수정 권장)
{정적 검사 위반 + 부패·깨진 참조}

### Warning (수동 판단)
{축별 목록}

### 안전 (개선 권장, 블로킹 아님)
{축별 목록}

### 다음 단계 제안
- (A) 이번 세션에서 즉시 정리
- (B) plan{N} 으로 task 화하여 실행
```

### 5. 사용자 승인 후 정리

**절대 사용자 승인 없이 docs 를 수정하지 않는다** — 일부 "bloat" 는 의도적 보존(마이그레이션 가이드 임시 등)일 수 있다.

승인된 항목만 순차 수정한다. 각 수정은 최소 diff — 결정 의도는 보존하되 표현·구조만 슬림화한다.
선택지를 낼 때는 구조화 질문 도구(Claude Code 는 `AskUserQuestion`)를 쓰고, 추천안은 첫 번째 + label 끝 `(추천)`.

## 정리 원칙 (수정 시 적용)

### ADR 표준 구조

```markdown
## ADR-XXX: {제목 — 결정의 한 줄 요약}

- **결정**: {무엇을 — 1~3문장}
- **맥락**: {왜 이 결정이 필요했는가 — 제약·데이터·관찰}
- **대안 기각**: {다른 옵션을 왜 선택하지 않았는가 — 각 대안 1~2줄}
- (선택) **트레이드오프**, **적용 범위**
```

금지:

- 15줄 넘는 코드 블록
- 파일 경로 3개 이상 나열
- 구현 호출 방법·컴포넌트 props 나열
- CLAUDE.md 스택 규칙 반복

### 문서 간 책임 분리 (단일 소스)

문서별 담당 정의는 오버레이의 docs 컨벤션 표를 따른다. 코어 원칙:

- **같은 내용은 한 문서에만** 둔다. 다른 문서는 `(ADR-XXX 참조)` 링크로.
- 기술 결정 근거는 ADR, 흐름은 flow, 스키마는 데이터 문서, 디렉터리·레이어는 아키텍처 문서.

### 새 ADR 작성 시 자명성 게이트

ADR 추가 전 아래를 자문한다:

1. **[필수] 자명성 통과**: 코드·설정·git log 로 같은 정보를 얻을 수 있으면 ADR 로 남기지 않는다.
2. **[필수] 유지 기준 해당**: 라이브러리 함정·실험 결과·대안 기각·정책·트레이드오프 중 하나.
3. **[필수] 3요소**: 결정·맥락·대안 기각 중 하나라도 비면 추론성 부족.
4. **[필수] 구현 세부 제거**: 코드 스니펫·경로 나열·작업 내역 제거.
5. **[필수] 인덱스 동기화**: 새 파일 생성 + Index 에 한 줄 추가.

### ADR 수정·폐기 시

- **의사결정 번복**: 대체 ADR 작성 + 과거 ADR 본문 **완전 삭제**(Superseded 표기 남기지 않음). 근거는 git log 와 새 ADR 본문에 녹인다.
- **부분 수용**: 기존 ADR 에서 해당 부분만 수정. 대안 기각에 "옵션 X 는 Y 이유로 번복" 한 줄 추가.
- **자명성 폐기**: 본문 + Index 동시 삭제. **폐기 번호는 결번으로 영구 보존하고 새 ADR 에 재할당하지 않는다** — 외부 참조(이슈·커밋·에이전트 메모리)가 과거 번호를 가리킬 때 다른 결정으로 오인되는 것을 막는다.

## 실행 주기 권장

| 시점 | 이유 |
|---|---|
| 대규모 구현 완료 후 | 코드 변경이 docs 에 반영됐는지 + 신규 ADR 이 bloat 없이 작성됐는지 |
| 외부 PR 머지 후 | docs 에 미반영된 변경 포함 가능 |
| 분기별 정기 | 점진적 부패·비대화 감지 |

## task 범위 검증과의 분담

구현 파이프라인 내부의 docs-verifier 는 **해당 task 범위만** 검증한다. 이 스킬은 **전체 docs** 를 6축으로 스캔한다.

| 도구 | 범위 | 시점 | 축 |
|---|---|---|---|
| 파이프라인 내 docs-verifier | 현재 task 변경 코드 ↔ docs | task 실행 직후 | A 부패 중심 |
| docs-check (이 스킬) | 전체 docs ↔ 전체 코드 + 품질 | 주기적·수동 | A~F 전체 |

## 의도적으로 안 하는 것

- **docs 수정·삭제**: 발견 전용 — 수정은 사용자 승인 후.
- **코드 변경**: docs 감사 전용. 코드 문제는 해당 task 로 처리.
- **ADR 자동 삭제**: 자명성 폐기 후보를 직접 삭제하지 않고 목록 제출 후 사용자 결정.
- **레포 특화를 코어에 하드코딩**: docs 구조·문서 목록·검증기 이름·grep 경로는 오버레이·CLAUDE.md 로만.
